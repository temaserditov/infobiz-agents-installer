# Аудит старой поддержки OpenClaw

## Источник и ограничения

Для аудита были агрегированно просмотрены 5 736 сообщений из технических разделов
двух прошлых потоков. Переписка использовалась только для поиска повторяющихся
сценариев. Имена, Telegram ID, вложения, токены и личные сообщения не включены в
документацию.

В прошлых потоках использовался OpenClaw. Его команды, каталоги, плагины, модели и
формат конфигурации не являются инструкцией для Hermes.

## Что создавало больше всего поддержки

### 1. Пользователь терял контекст терминала

Люди путали:

- локальный терминал и shell VPS;
- команду подключения и команду установки;
- browser callback и поле ввода в терминале;
- отсутствие символов при вводе пароля с зависанием;
- незапущенную строку с зависшей командой.

**Вывод:** каждый ручной переход между окнами кратно увеличивает ошибки. Для VPS
лучше browser console хостинга, device-code OAuth и одна команда установки.

### 2. Диагноз строился по скриншотам

На скриншотах часто отсутствовали начало ошибки, prompt, команда, время и финальный
exit status. Куратор вынужден был угадывать и иногда предлагал несовместимые шаги.

**Вывод:** продукт должен выдавать машинный redacted bundle. Codex должен читать
логи и сервисы напрямую.

### 3. Смешивались разные контуры

В одну категорию попадали:

- неверный пароль VPS;
- истёкший course token;
- ошибка загрузки;
- OAuth провайдера;
- недоступная модель;
- упавший gateway;
- неверный Telegram Bot Token.

**Вывод:** сначала определяется контур сбоя. Переустановка агента не лечит пароль
VPS, а новый course token не лечит gateway.

### 4. Зависимости ставились вручную

Повторялись ошибки npm permissions, несовместимых Node/Python, PATH, Homebrew и
сломанных venv.

**Вывод:** ученик не должен выбирать или вручную ставить runtime. Текущий продукт
сначала запускает официальный setup Hermes, затем добавляет управляемый Node.js для
WebShell и наш слой.

### 5. Gateway запускался не как управляемый сервис

После закрытия терминала, сна Mac, перезагрузки или сетевого сбоя бот переставал
отвечать. Ручные процессы дублировали `getUpdates` и конфликтовали за один токен.

**Вывод:** только launchd/systemd, один gateway на профиль, автоматический restart и
явная проверка health.

### 6. Модели и авторизация менялись вслепую

Ошибки `missing auth`, `unknown model`, timeout и provider failure лечились заменой
модели или ключа без проверки доступности аккаунту.

**Вывод:** модель выбирается из проверенного каталога WebShell; OAuth проверяется
отдельно; общий provider error расшифровывается по первичной строке gateway log.

### 7. Telegram настраивался частично

Токен сохранялся без проверки владельца, ID не ограничивался, gateway не всегда
перезапускался, один токен мог использоваться несколькими процессами.

**Вывод:** токен и allowlist ID — одна транзакция настройки; после сохранения нужен
автоматический restart и проверка статуса.

### 8. Повторный запуск ухудшал ситуацию

Пользователи тратили одноразовые попытки, запускали несколько установщиков,
создавали конкурирующие процессы и стирали полезный контекст ошибки.

**Вывод:** установка должна быть повторяемой, но повтор разрешается только после
классификации сбоя. Одноразовый token gate обязан отличать отказ доступа от ошибки
runtime.

## Что уже учтено в текущем Hermes-продукте

- официальный Hermes устанавливается до наших профилей;
- Python и Hermes venv создаются официальным setup;
- пользователь не вводит Telegram-токен в терминальном установщике;
- WebShell управляет токенами, Telegram ID, моделями и restart gateway;
- gateway запускаются через launchd/systemd;
- установка показывает минимум текста и отдельный OAuth-шаг;
- логи сохраняются в `~/InfobizAgents`;
- support bundle редактирует секреты;
- Mac и VPS имеют отдельные сервисные конфигурации;
- профили изолированы друг от друга.

## Что нельзя переносить из старой поддержки

- любые команды `openclaw ...`;
- пути `~/.openclaw/...`;
- ручные правки OpenClaw JSON и списков агентов;
- npm global install как способ ремонта Hermes;
- OpenClaw plugin/provider migration;
- старые gateway ports и LaunchAgent labels;
- копирование OpenClaw auth profiles между агентами.

## Требования к дальнейшим релизам

1. На каждом релизе проверять чистую установку Mac и Ubuntu VPS.
2. Проверять актуальный stable Hermes и совместимость с upstream `main`.
3. Тестировать повторный запуск, обновление и удаление.
4. Проверять пять gateway, WebShell, OAuth, модель, Telegram token и allowlist ID.
5. Не включать premium-скиллы и локальные данные в обычный payload.
6. Не выпускать локальный hotfix исходников Hermes без воспроизводимого патча и теста.
7. Сохранять пользовательские auth, `.env`, sessions, memory и документы при обновлении.
